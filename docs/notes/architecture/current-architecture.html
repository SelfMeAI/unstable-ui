unstable-ui 当前架构

这份文档记录当前已经确定的项目边界、核心抽象和非目标。后续实现可以演进，但默认都应在这里的框架内推进，而不是重新回到产品形态和技术路线发散。

1. 项目定位

unstable-ui 是一个给 agent / harness 使用的 React Native UI 框架。它不是 assistant 成品，不是 LLM 编排平台，也不是语音识别服务。它负责的是一层可嵌入的 UI 运行框架，让外部智能体把结构化界面渲染到移动端。

核心句子：任何远端或本地的 harness，只要输出符合协议的事件和 UI schema，这个框架就能在 React Native 中稳定地渲染和交互。

2. 核心原则

2.1 固定底盘，动态工作区

长期稳定的是框架底盘，例如语音入口、状态展示、错误与确认机制；任务工作区则由 harness 按协议描述并驱动。

2.1.1 默认不是聊天记录页

这个框架的默认首页不是传统对话列表，也不是连续聊天气泡。默认视图应该是当前工作区或当前任务页，历史对话只作为独立入口存在，用于回看和审计。

2.2 动态 UI 不是任意 UI

不允许服务端或模型直接下发 JSX、原生代码或任意脚本。所有动态界面必须通过受约束的 block schema 表达。

2.3 位置无关

agent 可以运行在云端、本地或混合环境。框架只依赖统一的 harness 接口和事件协议，不依赖 agent 的部署位置。

2.4 对外可嵌入，对内可扩展

对外以 npm 包形态发布，可以嵌入别人的 RN 页面；官方 demo 只是参考壳，不代表框架必须接管整个 app。

4. Harness 模型

  • Remote Harness：运行在服务端，通过 HTTP / SSE / WebSocket 等方式提供事件流。
  • Local Harness：运行在端内，直接通过 TS 接口回调事件。
  • Hybrid Harness：本地和云端混合，各自承担不同阶段能力。

不管哪种形态，UI 框架只认 HarnessAdapter。外部智能体用什么技术栈，不是当前仓库关心的对象。

5. 页面流转主线

当前代码和后续开发都应围绕一条统一主线推进，而不是把每个 screen 都当作彼此独立的静态页。

  1. 用户从底部固定 voice / text shell 发起输入。
  2. runtime 进入 processing 或 task 相关状态，并根据协议锁定输入或局部交互。
  3. harness 可以直接返回结果页，也可以先返回任务处理中间页，必要时再经过 finalizing 收束页。
  4. 任务处理中间页允许继续 patch、替换或推进到最终结果页，但主线默认优先明确的 staged screen。
  5. 最终结果页稳定后，输入重新解锁，用户可以继续发起下一轮。
后面的动画系统必须挂在这条页面流转主线上，而不是靠渲染层临时猜测“现在是不是该动一下”。

当前实现已经补上 renderer transition hooks。后续动画层默认只监听 screen / mode / flow / lock 的变化事件，不再自行推断业务流程。

6. Artifact 与 Capability

5.1 Artifact

agent 经常会产出链接、图片、音频、文件、HTML、PDF 等内容。框架不按来源区分，而是统一抽象成 artifact；来源可以是本地，也可以是远端。

5.2 Capability

设备能力如麦克风、相册、定位、文件选择、分享、外部打开，需要通过 capability request 走受控桥接，而不是让 agent 直接调原生 API。

7. 当前标准任务流

  • 输入 - processing - task page - finalizing - result。
  • 输入 - processing - direct result。
  • action - approval - result。
  • form - processing - finalizing - result。

之后新增 block、补动画、做远端示例时，都应先确认自己属于哪条标准流，避免再次回到“页面很多，但没有统一推进逻辑”的状态。

8. 非目标

  • 不做 LLM orchestration 平台。
  • 不做完整 agent framework。
  • 不做语音识别或模型服务本身。
  • 不让模型直接生成任意原生代码。
  • 不把 demo app 当成项目本体。
  • 不在仓库文档中继续写技术选型对比。
  • 不在页面流和交互锁未稳定前继续做重动画实验。

9. 当前实现状态

  • 仓库已经初始化为 npm workspace monorepo。
  • 已经补齐主包 @selfme/unstable-ui，作为默认安装入口。
  • GitHub Actions、Changesets、npm 发布骨架已经存在。
  • 协议包中已经有基础 schema、事件与 block 定义。
  • runtime 中已经有最小状态机和 provider。
  • renderer 已能渲染 text / card / list / actions / resource。
  • demo app 已用本地 harness 跑通最小闭环。
  • 语音入口壳、最小 artifact preview/open bridge、最小 capability request 流已实现。
  • 零依赖的 remote harness 示例服务已实现。
  • screen.mode、screen.interaction、history、输入锁和固定导航栏已经进入主干。
  • history 已经开始携带 request 级关联信息，能回看到底是哪一次输入、动作或表单触发了后续页面流。
  • runtime 已经补出 currentRequest、lastCompletedRequest 及其对应 history 视图，不再只剩扁平 event log。
  • request catalog、request resource query 和 request summary 这层 runtime helper 已正式导出，不再只存在 renderer 私有实现里。
  • renderer 已支持 runtime 派生的 actions source，request index 可以直接变成 drill-down 入口，而不是只显示不可点击的日志。
  • renderer 已支持 runtime 派生的 timeline source，request inspector 可以直接展示某条 request chain 的 stage replay 时间线。
  • renderer 已支持 runtime 派生的 stage summary source，request inspector 可以直接查看每个阶段的事件数、资源数、patch 数和 issue 数。
  • renderer 已暴露宿主级 imperative handle，宿主可以不经 harness screen，直接打开 history 或 request inspector overlay。
  • history / request inspector 现在已经进入正式导航事件模型，renderer 会发 navigation.changed，harness 可回发 navigation.updated 做同步或反向驱动。
  • history 分组现在可以直接点进 request inspector，inspector 自己也已经内建显式 requestId 钻取入口，开始替代 demo 专用的 inspector page。
  • runtime 已开始把 persistence 状态暴露成内建 source，可直接检查本地恢复、save 次数、最近一次保存时间和错误。
  • runtime 已开始把 transport 状态暴露成内建 source，并补出 reconnect / resetSession 这类正式宿主控制能力。
  • runtime 已开始把 recovery 断言和 recovery verdict 暴露成内建 source，可直接校验 restore / reconnect / reset / persistence 是否一致。
  • runtime 已开始把 artifact inventory、pending capability queue、bridge summary 以及 bridge assertions / verdict 暴露成内建可渲染 source。
  • renderer 已开始导出 bridge 接入 helper，如 createArtifactPreviewHandlers(...) 和 createCapabilityHandlers(...)，减少宿主手写样板代码。
  • renderer 现在也开始导出 createHostBridge(...)，把简单 host bridge 接入从手写对象推进到正式 helper。
  • renderer 现在还开始导出 createHostIntegration(...)，把 preview handler、capability preset 和 host bridge 组合成一个统一宿主接入对象。
  • renderer 已开始统一承担无 screen、connecting、error 和 in-flight 阶段态呈现，不再完全依赖 harness 自己补过渡说明。
  • screen lifecycle、runtime flow controller 和 renderer transition hooks 已进入主干。
  • screen lifecycle 已收成协议层派生语义：由 mode / flow 统一得出 root、transient、intermediate、final，避免再维护一组容易失真的可变状态。
  • demo 已经开始消费 transition hooks，过渡效果从框架事件驱动，而不是再靠页面内零散动画拼接。
  • 远端 harness 示例已经和本地 demo 对齐到明确的 request flow 页面阶段。
  • harness-sdk 已经开始导出正式的 screen flow helper，减少接入方自己拼 mode / flow / interaction。
  • timeline 类流程已经开始真实使用 screen.patched，而不是只做整屏替换。
  • demo 里已经有 Verification board，可基于请求链、patch 数、artifact 事件、会话恢复状态和 issue 数做派生校验。
  • 更完整的 artifact 渲染、真实设备桥接和更成熟的远端接入辅助仍未实现。
