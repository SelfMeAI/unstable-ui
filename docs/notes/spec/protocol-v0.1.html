unstable-ui 协议草案 v0.1

这一版协议以“先跑通最小闭环”为目标，覆盖当前仓库已经实现或已经明确准备支持的事件和数据结构。后续允许扩展，但不应随意破坏已有语义。

1. 协议总形态

协议是事件驱动的，不是一次性返回整页 HTML 或组件代码。harness 对客户端的主要输出是事件流，客户端对 harness 的主要输入也是事件。

  • harness 负责表达“此刻展示什么、等待什么、正在处理什么”。
  • 客户端负责状态管理、渲染和把用户操作回传成结构化事件。
  • 双方通过 JSON 交互，不传可执行 UI 代码。
  • request 级元数据要足够完整，方便 runtime 做历史串联和派生验证。
当前协议演进重点已经明确：优先把请求生命周期和页面流转语义补清楚，而不是继续堆更多视觉层变化。

4. Screen 语义

当前仓库已经引入 screen.mode、screen.flow 和 screen.interaction。screen 不再只是“长什么样”，还要表达“当前页是什么角色、属于哪轮请求、允许哪些交互、为什么被锁定”。

  • stable：稳定工作区，可继续输入或继续操作。
  • processing：请求刚进入时的处理中页面。
  • task：任务处理中间页，允许持续 patch 或被后续页替换。
  • result：最终结果页或阶段性结果页。
  • approval：等待用户确认或设备能力决策。
  • error：错误或恢复页。

screen.flow 里现在至少要能表达 requestId、ongoing / complete 和 replace / root。这是 runtime 管输入锁、历史和后续动画系统的共同基础。

另外，当前 demo 已经开始真实消费 screen.patched。同一个 request 内如果只是局部推进工作区，不应该总是整页替换，而应尽量用 patch 表达增量变化。

5.1 运行时派生验证视图

协议仍然只有一套事件流，没有单独再发一套“验证协议”。验证信息是 runtime 基于正常事件和 screen.flow 派生出来的。

  • runtime.currentRequest：当前活跃请求快照。
  • runtime.lastCompletedRequest：最近完成请求快照。
  • runtime.currentRequestHistory：当前请求关联 history。
  • runtime.lastCompletedRequestHistory：最近完成请求关联 history。
  • 请求指标：historyEntryCount、workspaceEventCount、patchEventCount、resourceEventCount、issueCount。
  • 导航调试源：runtime.navigation，可直接看到 history / request inspector overlay 当前是否打开、指向哪条 request。
  • 持久化调试源：runtime.persistence，可直接看到本地恢复是否命中、最近一次保存时间、load/save 状态与错误。
  • 传输调试源：runtime.transport，可直接看到 connect / reconnect / disconnect 次数、最近一次连接时间和连接错误。
  • 恢复调试源：runtime.recovery，可直接看到当前是 fresh / restored / reset / reconnected 哪种恢复模式，以及最近一次 reset / reconnect 时间。
  • 恢复断言源：runtime.recoveryAssertions，可直接得到 restore / reconnect / reset / persistence 这几类会话一致性检查。
  • 恢复 verdict 源：runtime.recoveryVerdict，可直接给出会话恢复、传输和持久化信号是否一致的总判断。
  • bridge 调试源：runtime.bridge、runtime.artifacts、runtime.capabilityRequests、runtime.capabilityHistory、runtime.lastCapabilityResolution。
  • bridge 集成源：runtime.bridgeIntegration，可直接查看宿主注册了哪些 host bridge 方法、哪些 artifact / capability handler 已经接入。
  • bridge 路由源：runtime.bridgeRouting，可直接看到代表性 artifact / capability 路径当前会走 artifact handler、host bridge、renderer default 还是 mock fallback。
  • bridge 错误源：runtime.bridgeErrors，可直接查看 renderer 侧 artifact / capability bridge 当前持有的错误信息。
  • bridge 断言源：runtime.bridgeAssertions，可直接检查 artifact inventory 是否有可操作路径、capability queue 与 resolution accounting 是否一致。
  • bridge verdict 源：runtime.bridgeVerdict，可直接给出 bridge-facing artifact 和 capability 信号是否一致的总判断。
  • 通用 request target 源：runtime.request、runtime.requestResources、runtime.requestHistory、runtime.requestResourceHistory、runtime.requestAssertions、runtime.requestMatrix、runtime.requestVerdict。
  • request 阶段摘要源：runtime.requestStageSummary 和 runtime.requestStageLog，可查看每个阶段的事件数、资源数、patch 数和 issue 数。
  • request 阶段回放时间线源：runtime.requestTimeline，可把某条 request chain 直接投影成 stage replay timeline。
  • request 级资源查询源：runtime.currentRequestResources、runtime.lastCompletedRequestResources、runtime.currentRequestResourceHistory、runtime.lastCompletedRequestResourceHistory。
  • request 索引源：runtime.requestIndexSummary 和 runtime.requestIndex。
  • request 动作源：runtime.requestIndexActions，可把 request catalog 直接变成可点击的钻取入口。
这意味着本地 harness 和远端 harness 只要老老实实发正常协议事件，就能自动获得同一套验证面，而不用额外实现第二套调试通道。
另外，这套 request 级派生能力现在也已经从 @selfme/unstable-ui-runtime 正式导出成 helper API， 自定义 renderer 或宿主调试页可以直接复用 request catalog、request target resolver、resource query 和 summary 逻辑，不需要再各自复制一份实现。
通用 request 源可以额外带一个 requestTarget。保留值只有 current 和 lastCompleted， 其他非空字符串都按显式 requestId 处理。
actions block 现在也可以走 runtime source。当前内建的是 runtime.requestIndexActions，用于把 request 索引直接映射成一组钻取动作。
timeline block 现在也可以走 runtime source。当前内建的是 runtime.requestTimeline，用于把单条 request chain 直接渲染成 stage replay 时间线。
details 和 log block 现在还能消费 runtime.requestStageSummary 与 runtime.requestStageLog， 用于查看 stage replay 背后的阶段指标。
协议现在还额外预留了框架自带 overlay 的导航事件： renderer 打开或关闭 history / request inspector 时会发 navigation.changed， harness 若要同步状态或直接驱动这些 overlay，可回发 navigation.updated。

6. 标准任务流

  • 输入 - processing - task page - finalizing - result。
  • 输入 - processing - direct result。
  • action - approval - result。
  • form - processing - finalizing - result。

后面 runtime、renderer、demo harness 和动画系统，都默认围绕这几条标准流展开，不再按孤立页面来理解协议。

7. Block 集合

当前内建 block 只保留最小集合，不追求一开始就完整。

  • text：纯文本块。
  • card：标题 + 正文概览块。
  • list：列表条目集合。
  • actions：一组操作按钮。
  • form：结构化输入块，提交后回传 form.submitted。
  • timeline：任务阶段时间线，适合表达 agent 进度与恢复点。
  • details：紧凑结构化摘要块，适合任务 brief、结果摘要和元信息。
  • log：日志流块，可直接映射 runtime event log 或 runtime history。
  • section：一层分组容器块，用来组织工作区。
  • split：双 pane 布局块，用来表达主次工作区。
  • resource：单个 artifact 资源块。

8. Artifact 预览元数据

现在 artifact 除了基础的 id / kind / uri / source，还可以附带轻量 preview 数据，供 renderer 默认预览层直接消费。

  • preview.text：适合 text / json 这类可直接展示的内容。
  • preview.summary：适合给默认预览层一句摘要。
  • preview.thumbnailUri：适合 image 这类资源提供一个缩略预览引用。
  • preview.fields：适合传一些结构化元信息。
这个字段不是让协议承载大文件本体，而是让 harness 在不自定义 host handler 的情况下，也能给默认 artifact preview 一个更像样的初始内容。

7. Screen Patch 操作

当前协议已经支持最小 screen patch，用来避免每次更新都整屏替换。

  • set_title：更新标题。
  • set_subtitle：更新副标题。
  • replace_blocks：整体替换 blocks。
  • append_blocks：向现有 blocks 末尾追加。
  • upsert_block：按 block id 更新或插入单个 block。
  • remove_block：按 block id 删除单个 block。
{
  "type": "screen.patched",
  "screenId": "home",
  "operations": [
    {
      "type": "set_subtitle",
      "subtitle": "Streaming additional blocks"
    },
    {
      "type": "append_blocks",
      "blocks": [
        {
          "id": "next-step",
          "type": "text",
          "value": "A new block arrived without replacing the full screen."
        }
      ]
    }
  ]
}

8. Artifact

Artifact 用来统一表达本地或远端产物。当前类型包括 text / image / audio / video / pdf / html / file / link / json。 

{
  "id": "artifact-1",
  "kind": "link",
  "uri": "https://github.com",
  "source": "remote",
  "title": "GitHub",
  "previewable": true,
  "openable": true
}

9. Capability

Capability request 用来请求设备能力。目前协议里已预留 microphone / camera / photo-library / location / file-picker / share / open-url。 

{
  "type": "capability.requested",
  "request": {
    "id": "req_1",
    "capability": "microphone",
    "reason": "需要录入语音输入"
  }
}

10. 当前判断

  • 第一版协议坚持“事件 + 受约束 block schema”，不回退到任意组件代码。
  • 当前已经有最小 artifact / capability / form 闭环，下一步重点先转向 lifecycle 和 flow controller。
  • 现在已经支持最小 screen patch，但仍然保持受约束的小操作集合，不开放任意 UI diff。
  • 现在已经把 screen.flow.requestId 作为 staged page flow、history 和 transition hooks 的主关联键。
  • interaction.history 现在已经是会真正影响 renderer 行为的控制位。
  • 当前已经具备基础 screen mode / interaction 语义，但还没有完成正式的页面流转协议。
  • 如果后续出现更复杂布局需求，优先扩展 block 和 layout schema，而不是引入可执行 UI。
