unstable-ui 协议草案 v0.1

这一版协议以“先跑通最小闭环”为目标，覆盖当前仓库已经实现或已经明确准备支持的事件和数据结构。后续允许扩展，但不应随意破坏已有语义。

1. 协议总形态

协议是事件驱动的，不是一次性返回整页 HTML 或组件代码。harness 对客户端的主要输出是事件流，客户端对 harness 的主要输入也是事件。

  • harness 负责表达“此刻展示什么、等待什么、正在处理什么”。
  • 客户端负责状态管理、渲染和把用户操作回传成结构化事件。
  • 双方通过 JSON 交互，不传可执行 UI 代码。
当前协议演进重点已经明确：优先把请求生命周期和页面流转语义补清楚，而不是继续堆更多视觉层变化。

2. Harness 事件

事件 作用 当前状态

status 更新运行阶段，例如 waiting、thinking、running 已实现
screen.updated 下发新的 screen schema 已实现
screen.patched 对当前 screen 做增量 patch 更新 已实现
artifact.available 声明某个 artifact 可被预览或打开 已实现

4. Screen 语义

当前仓库已经引入 screen.mode 和 screen.interaction。screen 不再只是“长什么样”，还要表达“当前页是什么角色、允许哪些交互、为什么被锁定”。

  • stable：稳定工作区，可继续输入或继续操作。
  • task：任务处理中间页，允许持续 patch 或被后续页替换。
  • result：最终结果页或阶段性结果页。
  • approval：等待用户确认或设备能力决策。
  • error：错误或恢复页。

下一步要补的不是新的基础类型，而是更正式的页面流转语义，例如当前页是 replace、临时任务页还是最终结果页，以及一轮请求结束后什么时候恢复输入。

5. 标准任务流

  • 输入 - processing - task page - result。
  • 输入 - processing - direct result。
  • action - approval - result。
  • form - processing - result。

后面 runtime、renderer、demo harness 和动画系统，都默认围绕这几条标准流展开，不再按孤立页面来理解协议。

6. Block 集合

当前内建 block 只保留最小集合，不追求一开始就完整。

  • text：纯文本块。
  • card：标题 + 正文概览块。
  • list：列表条目集合。
  • actions：一组操作按钮。
  • form：结构化输入块，提交后回传 form.submitted。
  • timeline：任务阶段时间线，适合表达 agent 进度与恢复点。
  • details：紧凑结构化摘要块，适合任务 brief、结果摘要和元信息。
  • log：日志流块，可直接映射 runtime event log。
  • section：一层分组容器块，用来组织工作区。
  • split：双 pane 布局块，用来表达主次工作区。
  • resource：单个 artifact 资源块。

7. Screen Patch 操作

当前协议已经支持最小 screen patch，用来避免每次更新都整屏替换。

  • set_title：更新标题。
  • set_subtitle：更新副标题。
  • replace_blocks：整体替换 blocks。
  • append_blocks：向现有 blocks 末尾追加。
  • upsert_block：按 block id 更新或插入单个 block。
  • remove_block：按 block id 删除单个 block。
{
  "type": "screen.patched",
  "screenId": "home",
  "operations": [
    {
      "type": "set_subtitle",
      "subtitle": "Streaming additional blocks"
    },
    {
      "type": "append_blocks",
      "blocks": [
        {
          "id": "next-step",
          "type": "text",
          "value": "A new block arrived without replacing the full screen."
        }
      ]
    }
  ]
}

8. Artifact

Artifact 用来统一表达本地或远端产物。当前类型包括 text / image / audio / video / pdf / html / file / link / json。 

{
  "id": "artifact-1",
  "kind": "link",
  "uri": "https://github.com",
  "source": "remote",
  "title": "GitHub",
  "previewable": true,
  "openable": true
}

9. Capability

Capability request 用来请求设备能力。目前协议里已预留 microphone / camera / photo-library / location / file-picker / share / open-url。 

{
  "type": "capability.requested",
  "request": {
    "id": "req_1",
    "capability": "microphone",
    "reason": "需要录入语音输入"
  }
}

10. 当前判断

  • 第一版协议坚持“事件 + 受约束 block schema”，不回退到任意组件代码。
  • 当前已经有最小 artifact / capability / form 闭环，下一步重点先转向 lifecycle 和 flow controller。
  • 现在已经支持最小 screen patch，但仍然保持受约束的小操作集合，不开放任意 UI diff。
  • 当前已经具备基础 screen mode / interaction 语义，但还没有完成正式的页面流转协议。
  • 如果后续出现更复杂布局需求，优先扩展 block 和 layout schema，而不是引入可执行 UI。
