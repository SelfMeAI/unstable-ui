unstable-ui Harness 接入说明

这份文档说明 React Native 主实现线当前支持的 harness 接入路径，以及建议的起步顺序。目标是让接入者先把最小闭环跑通，再逐步切到更真实的远端或混合接入形态。

1. 概览

unstable-ui 接受本地、远端和混合 harness，通过统一 adapter 模型接入。框架负责 React Native 侧的运行时、schema 渲染、artifact 处理和 capability 桥接；harness 负责输出事件与 screen schema。

核心原则：先统一协议面和接入接口，再区分 agent 实际运行在端内、云端还是混合环境。

2. 当前接入模式

2.1 createLocalHarness

  • 适合端内 mock 流程、本地任务编排和快速 UI 迭代。
  • 优点是闭环快、调试简单、无需先搭后端。
  • 适合作为协议、block 和运行时状态的第一调试入口。

2.2 createRemoteHarness

  • 适合 agent 或 workflow 运行在客户端之外的场景。
  • 当前推荐通过 HTTP + SSE 这类事件流方式把运行状态和 screen schema 推回端上。
  • 当事件流、screen schema 和 action 回流稳定后，再切到远端最合适。

2.3 Hybrid Harness

  • 适合一部分能力留在本地，一部分能力放在云端的场景。
  • 对框架来说，关键不是部署位置，而是是否继续遵守统一协议面。

3. 推荐起步路径

  1. 先用 @selfme/unstable-ui 作为默认安装入口。
  2. 先对接本地 harness，把最小屏幕渲染、action 回流和资源块跑通。
  3. 在 demo 里确认 block schema、artifact 行为和 capability 流没有歧义。
  4. 再切到远端 harness，接入真实服务端事件流。
先本地、后远端，不是能力限制，而是为了减少早期同时调协议、UI 和服务端的噪音。

4. Demo 参考

  • apps/demo-expo：演示本地 harness 路径、默认 runtime shell、Flow lab 调试入口、Runtime playground 页面和 Verification board 页面。
  • apps/demo-harness：演示最小远端 HTTP + SSE harness 服务，以及 direct / task / form / patched timeline 标准流，加上同样的 Runtime playground 和 Verification board 页面。

5. 派生 lifecycle 角色

协议现在提供 resolveScreenLifecycle(screen)。它不新增一组可变字段， 而是统一从已有的 mode 和 flow 推导页面意图， runtime 和 renderer 使用的是同一套判断。

  • root：不属于活跃请求的顶层稳定页面。
  • transient：harness 正在准备下一页时的 processing 页面。
  • intermediate：任务执行中或等待审批的中间工作页。
  • final：已完成的结果页或错误页。
后续动画只需要消费 renderer transition hook 里的 lifecycleRole，不应该再从任意 screen id 或 block 变化里猜测页面意图。

6. 页面流 helper

现在 harness-sdk 已经导出一组 screen helper，用来统一构建带 mode、flow 和 interaction 的页面。

  • createStableScreen
  • createProcessingScreen
  • createTaskScreen
  • createResultScreen
  • createErrorScreen
  • createRootScreenFlow
  • createOngoingScreenFlow
  • createCompletedScreenFlow
  • createFailedScreenFlow
  • applyScreenFlow
目的不是增加抽象层，而是减少接入者自己手拼页面状态字段，避免不同 harness 对 lifecycle 的表达越来越散。

7. 验证面

runtime 现在已经开始内建面向接入调试的验证视图，不需要 harness 额外发一套“验收事件”。

  • 按 requestId 串联的 history。
  • 当前请求和最近完成请求的派生快照。
  • 请求级 metrics：history、workspace、patch、resource、issue。
  • Verification board：断言列表、矩阵视图和总 verdict。
  • bridge 调试源：artifact inventory、pending capability queue、bridge summary，再加上 bridge integration、bridge routing、bridge errors、bridge assertions 和 bridge verdict。
  • request 级资源查询：当前请求和最近完成请求都能单独拉出 resource summary 和 resource history。
  • request 索引：现在还能把所有 request chain 汇总成一份可读索引，不必只盯当前或最近完成那两条。
  • request 索引动作：runtime 现在还能直接派生出一组可点击的 request drill-down actions，不用宿主自己手抄 requestId 再跳转。
  • request 阶段回放时间线：runtime 现在还能把单条 request chain 直接投影成 stage replay timeline，方便看阶段推进顺序。
  • request 阶段摘要：runtime 现在还能给每个阶段补出事件数、资源数、patch 数和 issue 数，便于快速定位哪一段有问题。
  • 持久化诊断：现在还能直接看到 runtime 是否从本地快照恢复、最近一次 save/load 状态和错误。
  • 传输诊断：现在还能直接看到连接次数、重连次数、最近一次连接时间和连接错误。
  • 恢复诊断：现在还能直接得到 restore / reconnect / reset / persistence 的断言列表和总 verdict。
接入方的重点不再是自己造一套 debug UI，而是先把 requestId、screen.mode、screen.flow、screen.patched 这些协议语义发准确。
终态 error screen 或 error event 会被识别为 failed flow，verdict 显示为 Failed。 这和 Needs review 不一样：前者说明任务确实失败，后者说明协议链路和预期生命周期不一致。
另外，这套 request 查询与汇总逻辑现在已经从 @selfme/unstable-ui-runtime 正式导出成 helper API， 自定义 renderer 或宿主调试面板可以直接复用 request catalog、request target resolver、resource query 和 summary 逻辑，不用再在各自工程里复制 renderer 内部实现。 开启 persistence 时，snapshot 也会保留最近完成和最近失败请求的 id，恢复后可以继续直接使用对应 inspector target。
默认 renderer 现在还额外暴露了一个宿主级 imperative handle。 宿主可以通过 AgentRuntimeViewHandle 直接调用 openHistory()、 openRequestInspector(target) 等方法， 不必一定等 harness 先发一个专门的 inspector screen。
如果 persistence adapter 实现了 clear()， 宿主现在也可以通过同一个 handle 直接调用 clearPersistence() 清掉本地 runtime 快照。
同一个 handle 现在也支持 reconnect() 和 resetSession()，可以直接从宿主触发运行时重连或整次会话重置。
这两个 overlay 现在也已经并入正式协议。 renderer 主动打开或关闭时会发 navigation.changed， harness 如果要同步状态或反向打开，也可以回发 navigation.updated。
默认 request inspector 现在已经内建显式 requestId 钻取入口， history 里的 request 分组卡片也可以直接点进 inspector，不必再额外发一整页专门的 inspector screen。
如果宿主还需要知道 overlay 当前在看哪条 request，或 history 是否已经展开， 现在还可以传 navigationHooks，其中包含 onRequestInspectorChange 和 onHistoryVisibilityChange。

7. 当前判断

  • React Native 主实现线现在已经具备最小但真实的接入闭环。
  • 当前重点是让本地 / 远端 / 混合三类接入都能落在统一 adapter 模型上。
  • 默认 renderer 已经内建 open-url 和 share 两类系统桥接，不必一开始就自定义 handler。
  • 默认 renderer 也已经内建 artifact 的 share 和 download 行为，资源块不再只停留在 preview / open。
  • history 现在已经开始绑定 requestId，同一次输入触发出来的 processing / task / result 页面能被串起来回看。
  • runtime 已经能派生 currentRequest、lastCompletedRequest 和请求级验证指标；最近失败请求也可以通过 lastFailed target 直接打开 inspector，调试重点正在从“看零散 event log”转向“看整条 request 链”。
  • interaction.history 现在是正式生效的 renderer 控制位，不再只是协议里的摆设字段。
  • 如果宿主只想接管系统打开、分享或默认 capability resolution，而不想重写更高层 handler，现在可以走 hostBridge。
  • 如果宿主只想补 openUrl、share 或简单 capability payload 覆盖，可直接用 createHostBridge(...)，不用手写完整 resolveCapability()。
  • 如果宿主只是想按资源种类补 preview，可直接用 createArtifactPreviewHandlers(...)，不用手写整套 handler 结构。
  • 如果宿主只是想补 capability prompt 和简单 payload 覆盖，可直接用 createCapabilityHandlers(...)，不用再手写完整 resolve()。
  • 如果宿主想把 preview override、capability preset 和 host bridge wiring 一次组合好，可直接用 createHostIntegration(...)。
  • 如果宿主只想调输入壳文案、默认模式和最近输入行为，可以先走 voiceShell 配置；只有要完全替换底部输入 UI 时才需要上 renderVoiceShell。
  • 它现在更适合 starter integration，而不是完整生产接入套件。
