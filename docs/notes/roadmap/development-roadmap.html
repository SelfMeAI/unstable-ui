unstable-ui 开发路线

这份路线不是对外 roadmap，而是当前仓库实际建议的推进顺序。目的是减少来回跳题，优先把协议、基础壳和桥接能力跑通。

1. 已完成

  • 初始化 monorepo。
  • 拆出 protocol、harness-sdk、runtime、core-blocks、renderer、demo-expo。
  • 补齐主包 @selfme/unstable-ui，作为默认安装入口。
  • 补齐 GitHub Actions、Changesets、npm 发布骨架。
  • 实现最小协议和最小本地 harness 闭环。
  • 建立内部中文 HTML 文档目录。
  • 实现固定 voice shell v0 与最小 artifact preview/open bridge。
  • 增加零依赖的远端 harness 示例服务。
  • 实现最小 capability request / capability.resolved 闭环。
  • 增加内建 form block，并打通 form.submitted 闭环。
  • 增加内建 timeline block，并打通远端/本地 staged progression 示例。
  • 增加内建 details block，用于任务摘要、结果元信息和结构化 brief。
  • 增加内建 log block，并把 runtime event log 真正映射进工作区视图。
  • 增加内建 section block，把单列堆叠提升为可分组的 workspace 组织方式。
  • 增加内建 split block，让工作区具备主次 pane 布局能力。
  • 增加 session history 记录，并在 renderer 顶部提供固定 History 入口。
  • 把底部输入壳扩展为 voice / text 双模式快速切换。
  • 在 renderer 暴露页面过渡钩子，为后续动画系统提供正式挂点。
  • 在 demo 中接入基于 transition hooks 的全屏过渡控制器，先把 processing / settling 链路跑通。
  • 把远端 harness 示例收敛到明确的 processing / task / finalizing / result 页面流。
  • 把 screen flow helper 提炼进 harness-sdk，减少接入方手拼 lifecycle 字段。
  • 把 runtime 的 request-aware history、request metrics 和 Verification board 主干补齐。
  • 让 timeline 流真正走 screen.patched 增量更新，验证 patched request 的运行语义。
  • 把 renderer 的无 screen / connecting / error / in-flight 阶段态补成统一的框架级状态呈现，不再只剩零散裸文本。
  • 把 bridge integration / routing / errors / assertions / verdict 收成正式 runtime source，并开始导出 bridge helper API，减少宿主桥接接入样板代码。
  • 把 bridge helper API 再向上收成 createHostIntegration(...) 统一 builder，减少宿主在 artifact/capability/hostBridge 三层之间来回组装。

2. 当前下一阶段：先把页面流转打牢

现在不继续纠缠动画，也不继续做零散视觉实验。下一步直接进入页面流转主干建设，把“输入后经历什么页面、什么时候锁输入、什么时候回到可交互状态”这套机制定死。

  1. 先补 screen lifecycle 语义，明确 idle、processing、task、result、approval、error 这些页面角色。
  2. 再补 runtime flow controller，明确当前 screen 是替换、临时过渡、任务中间态还是最终结果态。
  3. 在 renderer 层固定 transition hooks，让动画系统只消费过渡事件，不再反向侵入 runtime 语义。
  4. 把 demo harness 收敛成几条标准流程，而不是继续堆随机页面。
  5. 等页面流、输入锁和历史记录都稳定后，再重新设计动画系统。
当前判断：动画效果做不出来，不是单纯动效参数问题，而是页面生命周期和过渡挂点还不够清晰。

3. 阶段一：把框架主干跑通

  1. 把固定 input shell 与页面流真正绑定起来，形成稳定的提交、锁定、解锁闭环。
  2. 把 history 从附属能力提升为正式会话入口，保证能回看输入、任务页、结果页和错误页。
  3. 继续把 history 做成 request-aware，并补上 current / completed request 级别的验证视图。
  4. 增强 artifact bridge：从当前最小 preview/open 扩展到更真实的资源预览行为。
  5. 增强 capability bridge：从当前最小 mock 流扩展到真实设备权限和系统能力桥接。
  6. 把 renderer 的状态显示、空态、错误态、processing 态和 verification 呈现再做完整一点。

4. 阶段二：把接入面补齐

  1. 把 remote harness 官方示例继续从 demo 提炼成更稳的对外接入范式。
  2. 固化推荐的 HTTP / SSE 页面流约定与 requestId 传递方式。
  3. 补本地 mock / playground，便于协议调试。
  4. 继续扩 block 集合，但优先顺序要服从页面流和接入稳定性。

5. 阶段三：增强稳定性与可扩展性

  1. 把 streaming patch 从 demo 级验证推进到更稳定的正式接入能力。
  2. 继续增强会话恢复、本地缓存、event log 和 history 查询能力。
  3. 把 artifact preview 做成可扩展注册机制。
  4. 整理更正式的对外接入文档和示例仓库。

6. 标准任务流样例

  • 语音输入 - processing - task page - finalizing - result。
  • 语音输入 - processing - direct result。
  • action.triggered - approval - result。
  • form.submitted - processing - finalizing - result。

这几条流程后面会作为 runtime、renderer、demo harness 和动画系统的共同基线。

7. 阶段验收标准

5.1 阶段一

  • demo 中能看到固定壳 + 动态工作区。
  • 能触发 action 并收到协议回流。
  • 至少一种 artifact 能 preview/open 跑通。
  • 输入锁、返回首页、历史回看和标准任务流都稳定可复现。

5.2 阶段二

  • 远端 harness 能通过官方示例接入。
  • 外部开发者能看着示例接出自己的最小后端。

5.3 阶段三

  • 协议和运行时具备基本扩展能力，不再只是 demo 级别。
  • 框架有能力承载更复杂的任务流，而不需要推翻分层。
