unstable-ui 开发路线

这份路线不是对外 roadmap，而是当前仓库实际建议的推进顺序。目的是减少来回跳题，优先把协议、基础壳和桥接能力跑通。

1. 已完成

  • 初始化 monorepo。
  • 拆出 protocol、harness-sdk、runtime、core-blocks、renderer、demo-expo。
  • 补齐 GitHub Actions、Changesets、npm 发布骨架。
  • 实现最小协议和最小本地 harness 闭环。
  • 建立内部中文 HTML 文档目录。
  • 实现固定 voice shell v0 与最小 artifact preview/open bridge。
  • 增加零依赖的远端 harness 示例服务。
  • 实现最小 capability request / capability.resolved 闭环。

2. 阶段一：把框架主干跑通

  1. 做最小 voice shell：固定入口、状态栏、按住说话交互占位。
  2. 增强 artifact bridge：从当前最小 preview/open 扩展到更真实的资源预览行为。
  3. 增强 capability bridge：从当前最小 mock 流扩展到真实设备权限和系统能力桥接。
  4. 把 renderer 的状态显示、空态、错误态再做完整一点。

3. 阶段二：把接入面补齐

  1. 实现 remote harness 官方示例。
  2. 定义推荐的 HTTP / SSE 接入形态。
  3. 补本地 mock / playground，便于协议调试。
  4. 把 block 扩到下一组常用集合，例如 form、sheet、timeline。

4. 阶段三：增强稳定性与可扩展性

  1. 支持 streaming patch，而不只是整屏更新。
  2. 加入会话恢复、本地缓存和 event log。
  3. 把 artifact preview 做成可扩展注册机制。
  4. 整理更正式的对外接入文档和示例仓库。

5. 阶段验收标准

5.1 阶段一

  • demo 中能看到固定壳 + 动态工作区。
  • 能触发 action 并收到协议回流。
  • 至少一种 artifact 能 preview/open 跑通。

5.2 阶段二

  • 远端 harness 能通过官方示例接入。
  • 外部开发者能看着示例接出自己的最小后端。

5.3 阶段三

  • 协议和运行时具备基本扩展能力，不再只是 demo 级别。
  • 框架有能力承载更复杂的任务流，而不需要推翻分层。
