unstable-ui 仓库分层

这份文档说明当前 monorepo 的目录设计和职责边界，避免后续开发时把 demo、协议、渲染器和 runtime 又混回一个包里。

2. package 职责

2.1 @selfme/unstable-ui-protocol

  • 只放 schema、类型、解析器和协议常量。
  • 不放 React、RN 视图或平台相关实现。

2.2 @selfme/unstable-ui-harness-sdk

  • 只放 harness adapter 抽象与官方 adapter 工具。
  • 远端、本地、混合接入优先从这里扩展。

2.3 @selfme/unstable-ui-runtime

  • 负责会话状态机、provider、hooks、生命周期。
  • 不直接负责视觉表现。

2.4 @selfme/unstable-ui-core-blocks

  • 放官方默认 block 的实现。
  • 后面 block 增长后，这里仍然只保留“基础官方集”。

2.5 @selfme/unstable-ui-renderer

  • 把 runtime 和 core blocks 组织成默认 RN 视图。
  • 对外提供 AgentRuntimeView 这类可嵌入组件。

3. demo 的定位

  1. demo-expo：验证协议和渲染器是否能跑通闭环，并展示默认 agent shell。
  2. demo-harness：提供零依赖的远端 harness 示例服务，演示 HTTP + SSE 接入形态。
  3. 两者一起构成最小本地 / 远端参考实现。
任何只对 demo 有意义、但不对框架本身成立的逻辑，都不应该沉到 package 里。

4. 发布链路

  • 根目录使用 npm workspace 管理依赖。
  • 发布版本管理使用 Changesets。
  • CI 和 release workflow 已经存在。
  • 每个发布包都有 prepack，发布前会先 build。

5. 后续维护规则

  • 协议改动优先改 protocol，不要先在 renderer 里偷偷扩展。
  • 涉及运行状态的新能力优先落在 runtime。
  • 如果只是视觉 block，优先落在 core-blocks。
  • demo 可以做演示，但不要成为真实逻辑的唯一入口。
